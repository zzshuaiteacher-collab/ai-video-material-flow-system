# Skill 沉淀规则

本文件用于判断一个流程是否应该沉淀为 skill。

## 1. 总原则

不要一发现流程就立刻写 skill。

先判断它是否稳定、可复用、可迁移。

规则：

```text
先评估
→ 后确认
→ 再创建或更新 skill
```

## 2. 适合沉淀为 skill 的条件

一个流程同时满足以下多数条件时，可以建议 skill 化：

1. 会反复出现。
2. 输入和输出稳定。
3. 判断标准清楚。
4. 不绑定某个单一品牌或单一文件夹。
5. 可以迁移到其他项目。
6. 有明确触发语。
7. 可以减少总控重复解释。
8. 已经至少跑通过一次。

## 3. 不适合过早 skill 化的情况

以下情况先不要写 skill：

- 只做过一次，还不稳定。
- 依赖大量临时判断。
- 只适用于一个项目。
- 用户还没确认。
- 输入输出还经常变化。
- 还没有复盘问题。

## 4. Skill 候选记录格式

每个 Agent 可以在执行记录中写：

```text
是否建议 skill 化：待评估

skill 化建议：
这个流程可能适合沉淀为 <skill-name>。
原因：
1. ...
2. ...
建议等再跑一轮后由总控确认。
```

## 5. 总控评估格式

总控评估时使用：

```text
流程名称：
是否适合 skill 化：
判断理由：
适用范围：
输入：
输出：
触发语：
是否需要用户确认：
建议：
```

## 6. 常见素材流 skill 候选

| 候选 skill | 适用场景 |
| --- | --- |
| 素材流整理 | 扫描、命名、分类、归档、新增素材处理 |
| 参考素材分析与写作标准 | 拆镜头、节奏、字幕、模板结构并形成写作规范 |
| 内容生产任务单 | 根据素材和模板生成剪辑任务单 |
| 视频生成验收 | 检查视频参数、字幕、关键帧、生成记录 |
| 合规自检 | 字幕、画面、音乐、版权、隐私风险检查 |
| 发布归档复盘 | 发布记录、数据回填、效果复盘 |

## 7. 用户确认门

只有用户明确说：

```text
可以
确认
就按这个写成 skill
```

总控才可以创建或更新 skill。

用户只是问“能不能写成 skill”，不等于授权创建。
