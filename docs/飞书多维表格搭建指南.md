# 飞书多维表格搭建指南

本文件用于把本项目的通用协作表字段，落地到飞书多维表格。

如果你使用 Notion、Airtable、Google Sheet，也可以参考字段含义迁移；但本文件会按飞书多维表格的字段类型和视图来写。

## 0. 总控前置检查

本系统默认在第一条视频任务开始前完成飞书搭建。总控先执行：

```powershell
Get-Command lark-cli -ErrorAction SilentlyContinue
lark-cli --version
```

若未安装，并且设备已有 Node.js 与 `npm`，安装：

```powershell
npm install -g @larksuite/cli
```

安装包地址：

```text
https://www.npmjs.com/package/@larksuite/cli
```

授权不要让当前会话一直等待：

1. 后台启动 `lark-cli config init --new`，把它返回的授权链接原样交给用户。
2. 配置完成后运行 `lark-cli auth login --recommend --no-wait --json`。
3. 将返回的 `verification_url` 原样交给用户，保留 `device_code`。
4. 用户完成授权后运行 `lark-cli auth login --device-code <device_code>`。

## 1. 搭建原则

首条完整 MVP 前只建 3 张表：

1. 视频任务总表
2. 素材与模板表
3. Agent 执行记录表

不要一开始创建太多表。先让系统跑通，再根据复盘拆表。

## 2. 建表顺序

建议按这个顺序建：

```text
视频任务总表
→ 素材与模板表
→ Agent 执行记录表
→ 视图
→ 可选自动化
```

## 3. 视频任务总表

用途：管理每条视频从选题、素材、剪辑、审核、发布到复盘的主状态。

### 3.1 字段配置

| 字段名 | 飞书字段类型 | 推荐选项或说明 |
| --- | --- | --- |
| 视频编号 | 文本 | 例如 `V001`、`V002` |
| 内容主题 | 文本 | 本条视频主题 |
| 内容类型 | 单选 | 产品展示、探店、口播、课程、门店氛围、其他 |
| 当前状态 | 单选 | 待生产、素材准备、剪辑中、AI初审、人工终审、待发布、已发布、已归档、返工 |
| 当前负责人/Agent | 单选 | 总控Agent、素材与模板Agent、内容生产Agent、审核发布Agent、人工 |
| 生成视频路径 | 文本 | 本地路径、云盘链接或附件说明 |
| AI初审结论 | 单选 | 未审核、通过、需修改、不通过 |
| 人工终审结论 | 单选 | 未审核、通过、需修改、不通过 |
| 发布状态 | 单选 | 未发布、待发布、已发布、已归档 |
| 发布链接 | URL 或文本 | 如果飞书 URL 字段不便复制，可用文本 |
| 发布链接类型 | 单选 | 真实发布、占位测试、竞品参考 |
| 发布时间 | 日期 | 建议开启具体时间 |
| 风险摘要 | 多行文本 | AI 初审和人工确认风险 |
| 最终人工决策说明 | 多行文本 | 人工通过、修改或不通过原因 |
| 数据回填状态 | 单选 | 未开始、待回填、已回填、无需回填 |
| 复盘结论 | 多行文本 | 本轮复盘摘要 |
| 下一步动作 | 多行文本 | 当前需要谁做什么 |
| 备注 | 多行文本 | 其他说明 |

### 3.2 推荐视图

| 视图名 | 视图类型 | 筛选条件 | 用途 |
| --- | --- | --- | --- |
| 总控看板 | 表格或看板 | 当前状态 不等于 已归档 | 总控每天看所有未收口任务 |
| 待素材处理 | 表格 | 当前负责人/Agent = 素材与模板Agent | 素材 Agent 领取任务 |
| 待内容生产 | 表格 | 当前负责人/Agent = 内容生产Agent | 内容生产 Agent 领取任务 |
| 待审核发布 | 表格 | 当前负责人/Agent = 审核发布Agent | 审核发布 Agent 领取任务 |
| 待人工确认 | 表格 | 当前负责人/Agent = 人工 或 当前状态 = 人工终审 | 用户确认 |
| 已归档 | 表格 | 当前状态 = 已归档 | 历史记录 |
| 数据待回填 | 表格 | 数据回填状态 = 待回填 | 发布后数据追踪 |

### 3.3 状态流转

推荐状态流：

```text
待生产
→ 素材准备
→ 剪辑中
→ AI初审
→ 人工终审
→ 待发布
→ 已发布
→ 已归档
```

如果需要返工：

```text
返工
→ 回到对应 Agent
```

## 4. 素材与模板表

用途：管理素材、音乐、参考视频、模板和风险状态。

### 4.1 字段配置

| 字段名 | 飞书字段类型 | 推荐选项或说明 |
| --- | --- | --- |
| 资产编号 | 文本 | 例如 `A001` |
| 资产名称 | 文本 | 文件名、素材名或模板名 |
| 资产类型 | 单选 | 原始视频、已整理素材、音乐、参考视频、模板、风险素材 |
| 分类 | 文本 或 单选 | 制作、成品、环境、顾客、包装、口播、待分类等 |
| 路径或链接 | 文本 或 URL | 本地路径、云盘链接或附件说明 |
| 可用状态 | 单选 | 可用、禁用、待确认、待分类 |
| 风险标签 | 多选 | 顾客正脸、订单、小票、支付码、手机号、版权、价格、地址、活动 |
| 适用模板 | 文本 | 可用于哪些视频模板 |
| 备注 | 多行文本 | 素材说明 |
| 最后更新 | 日期 | 更新时间 |

### 4.2 推荐视图

| 视图名 | 筛选条件 | 用途 |
| --- | --- | --- |
| 可用素材 | 可用状态 = 可用 | 内容生产 Agent 选素材 |
| 待确认素材 | 可用状态 = 待确认 | 用户或总控确认 |
| 风险素材 | 风险标签 不为空 | 审核发布 Agent 关注 |
| 待分类素材 | 可用状态 = 待分类 或 分类 包含 待分类 | 素材 Agent 处理 |
| 音乐库 | 资产类型 = 音乐 | 选音乐和版权确认 |
| 参考视频 | 资产类型 = 参考视频 | 参考拆解 |
| 模板库 | 资产类型 = 模板 | 内容生产复用 |

## 5. Agent 执行记录表

用途：记录每次 Agent 执行了什么、产出了什么、遇到什么问题、是否建议 skill 化。

### 5.1 字段配置

| 字段名 | 飞书字段类型 | 推荐选项或说明 |
| --- | --- | --- |
| 执行编号 | 文本 或 自动编号 | 例如 `E001` |
| 关联视频编号 | 文本 | 对应视频任务编号 |
| 执行时间 | 日期 | 建议开启具体时间 |
| Agent | 单选 | 总控Agent、素材与模板Agent、内容生产Agent、审核发布Agent |
| 任务类型 | 单选 | 系统初始化、素材整理、模板拆解、内容生产、合规审核、发布归档、数据回填、复盘、skill建议 |
| 使用模型 | 文本 或 单选 | 按实际平台填写 |
| 执行状态 | 单选 | 待执行、执行中、已完成、需人工确认、失败 |
| 输入来源 | 多行文本 | 读取了哪些文件、表格或链接 |
| 输出产物 | 多行文本 | 生成了哪些文件或记录 |
| 问题记录 | 多行文本 | 遇到的问题 |
| 是否建议skill化 | 单选 | 否、是、待评估 |
| skill化建议 | 多行文本 | 建议沉淀的能力 |

### 5.2 推荐视图

| 视图名 | 筛选条件 | 用途 |
| --- | --- | --- |
| 全部执行记录 | 无 | 总记录 |
| 失败与问题 | 执行状态 = 失败 或 问题记录 不为空 | 总控排查 |
| 需人工确认 | 执行状态 = 需人工确认 | 用户处理 |
| Skill 候选 | 是否建议skill化 = 是 或 待评估 | 总控定期评估 |
| 今日执行 | 执行时间 = 今天 | 日常追踪 |

## 6. 字段类型注意事项

### 6.1 单选字段

状态、负责人、结论这类字段优先使用单选。

好处：

- 方便筛选
- 方便做看板
- 减少 Agent 写错状态

### 6.2 多选字段

风险标签适合用多选。

同一个素材可能同时有：

```text
顾客正脸 + 小票 + 手机号
```

### 6.3 长文本字段

复盘、风险摘要、问题记录、skill 建议建议用多行文本。

不要把这些信息都塞进备注，否则后续很难筛选。

### 6.4 附件字段

MVP 阶段不强制使用附件字段。

原因：

- 附件上传和下载对 Agent 更复杂
- 不同平台权限容易出问题
- 先用路径或链接更稳定

等流程稳定后，再考虑给视频、关键帧、审核报告增加附件字段。

## 7. Agent 写入规则

如果 Agent 通过命令行或 API 写入飞书，多语言字段容易遇到编码问题。

推荐规则：

1. 不要直接在命令行里内联大段中文 JSON。
2. 先写入临时 UTF-8 JSON 文件。
3. 再用平台工具读取该 JSON 文件提交。
4. 写入完成后删除临时文件。
5. 不要把临时 JSON 文件提交到 GitHub。

`.gitignore` 中已经排除了常见临时目录：

```text
.lark-temp/
outputs/**/tmp_upserts*/
```

### 7.1 推荐写法（用 `--json @file` 避免中文乱码）

如果你用 `lark-cli` 写入 Base（多维表格），优先用 `--json @<file_path>` 从文件读取 JSON，而不是把中文 JSON 直接写在命令行里。

示例（PowerShell，创建一个临时 JSON 文件）：

```powershell
New-Item -ItemType Directory -Force .lark-temp | Out-Null

$payload = @'
{"fields":["视频编号","当前状态","复盘结论"],"rows":[["V001","已归档","本轮仅验证流程闭环"]]}
'@

$utf8NoBom = [System.Text.UTF8Encoding]::new($false)
[System.IO.File]::WriteAllText((Join-Path (Get-Location) '.lark-temp\batch-create.json'), $payload, $utf8NoBom)
```

然后提交（示例命令，token/id 均用占位符，按实际替换）：

```powershell
lark-cli base +record-batch-create --base-token <base_token> --table-id <table_id> --json "@.lark-temp/batch-create.json"
```

更新同理（用 `+record-batch-update`）：

```powershell
lark-cli base +record-batch-update --base-token <base_token> --table-id <table_id> --json "@.lark-temp/batch-update.json"
```

完成后清理临时文件：

```text
删除 .lark-temp/ 目录下的临时 json
```

Windows PowerShell 注意事项：

- `--json @.lark-temp/file.json` 会被 PowerShell 当成自己的语法；要写成字符串参数 `--json "@.lark-temp/file.json"`。
- Windows PowerShell 的 `Set-Content -Encoding UTF8` 可能写入 BOM，导致 CLI 报 JSON 开头包含非法字符；使用上面的 `UTF8Encoding($false)` 写法生成无 BOM 文件。
- 授权链接直接发给用户打开即可；不要把二维码工具作为安装流程的必要依赖。

## 9. 后续扩展

当项目稳定后，可以考虑拆出新表：

| 新表 | 什么时候拆 |
| --- | --- |
| 发布数据表 | 当真实发布数量变多，需要记录播放、点赞、评论、完播率 |
| 审核记录表 | 当审核问题很多，需要单独追踪风险 |
| 素材库明细表 | 当素材数量很大，需要更细标签 |
| 模板库表 | 当模板数量很多，需要版本管理 |
| Skill 候选表 | 当多个项目同时产生 skill 建议 |

MVP 阶段先不要拆太早。
